# FRC Tech Stack and Best Practices (2026 Season)

**CRITICAL INSTRUCTION FOR AI AGENTS:** This document is the absolute source of truth for all software development in this repository. You MUST adhere to these rules strictly. Do not hallucinate deprecated WPILib methods, old vendor libraries, or non-AdvantageKit architectures.

## 1. Core Technology Stack
* **Language:** Java 17+
* **Framework:** WPILib 2026 (Modern Command-Based Framework)
* **Telemetry & Logging:** AdvantageKit (v3+)
* **Drivetrain:** CTRE Tuner X Swerve Generator (Phoenix 6 Swerve API)
* **Motors & Hardware:** * CTRE Phoenix 6 API (for Falcons, Krakens, CANcoders, Pigeon 2)
    * REVLib API (for NEOs, Vortex, SparkMax, SparkFlex)
* **Mechanisms:** YAMS (Yet Another Mechanism System)
* **Sim Vision:** PhotonVision
* **Vision:** Limelight
* **Testing:** JUnit 5

## 2. Drivetrain (CTRE Tuner X Exception)
* **The Exception:** The Swerve Drivetrain is generated by CTRE Tuner X. It DOES NOT follow the standard AdvantageKit 3-file IO pattern. **DO NOT attempt to rewrite the drivetrain into `SwerveModuleIO` interfaces.**
* **Control:** Always control the swerve drive using CTRE's `SwerveRequest` objects (e.g., `SwerveRequest.FieldCentric`, `SwerveRequest.RobotCentric`).
* **AdvantageKit Logging for Swerve:** Log the drivetrain by fetching the state from the CTRE `SwerveDrivetrain` class and passing it to AdvantageKit's logger.
    ```java
    // Inside CommandSwerveDrivetrain periodic()
    SwerveDriveState state = this.getState();
    Logger.recordOutput("Swerve/Pose", state.Pose);
    Logger.recordOutput("Swerve/ModuleStates", state.ModuleStates);
    Logger.recordOutput("Swerve/ModuleTargets", state.ModuleTargets);
    ```
* **Simulation:** CTRE Phoenix 6 handles its own swerve simulation natively. Do not write a custom `SwerveIOSim`.

## 3. AdvantageKit Architecture (The IO Pattern)
For ALL subsystems *except* the Swerve Drivetrain, we use AdvantageKit for deterministic logging and replay. **Hardware must NEVER be instantiated directly inside a Subsystem.**

### The 3-File Subsystem Rule
Every standard subsystem must be split into at least three files:
1.  **`[SubsystemName]IO.java`**: The hardware interface.
    * Must contain an `@AutoLog` static nested class named `[SubsystemName]IOInputs`.
    * Must contain a default method `updateInputs([SubsystemName]IOInputs inputs)`.
2.  **`[SubsystemName]IOHardware.java`**: The real hardware implementation.
    * Implements the `IO` interface.
    * This is the ONLY place where `TalonFX`, `CANSparkMax`, or `DoubleSolenoid` objects are created.
3.  **`[SubsystemName].java`**: The WPILib Subsystem logic.
    * Extends `SubsystemBase`.
    * Accepts the `IO` interface in its constructor.
    * Instantiates the autologged inputs object: `private final [SubsystemName]IOInputsAutoLogged inputs = new [SubsystemName]IOInputsAutoLogged();`
    * In the `periodic()` method, ALWAYS call:
        ```java
        io.updateInputs(inputs);
        Logger.processInputs("Subsystems/[SubsystemName]", inputs);
        ```

## 4. Mechanisms (YAMS Integration)
* **What to use it for:** Use YAMS (Yet Another Mechanism System) for all elevators, arms, turrets, and moving joints. DO NOT write custom WPILib `ProfiledPIDController` or `ArmFeedforward` math from scratch.
* **Anti-Hallucination Rule:** The `Researcher` and `Implementer` agents MUST use tools to look up the official YAMS GitHub repository (`Yet-Another-Software-Suite/YAMS`) or documentation BEFORE attempting to write YAMS implementation code.

---

## 5. Vendor Library Strict Rules

### CTRE (Phoenix 6 ONLY)
* **NEVER** use Phoenix 5 or deprecated classes (e.g., `WPI_TalonFX`, `TalonSRX`).
* **ALWAYS** use the `com.ctre.phoenix6` package.
* Instantiate motors using `new TalonFX(deviceID, "canbus_name")`.
* Use Phoenix 6 Control Requests (e.g., `VoltageOut`, `PositionVoltage`) instead of `.set()`.

### REV Robotics
* Use `CANSparkMax` or `CANSparkFlex` depending on the hardware.
* Always configure motors using the builder pattern or `restoreFactoryDefaults()`.

## 6. Modern Command-Based Practices
* **No Command Classes:** Avoid creating separate `.java` files for Commands. Use inline Command factories inside the Subsystem.
* **Example Factory Method:**
    ```java
    public Command positionCommand(double target) {
        return this.run(() -> io.setPosition(target))
                   .finallyDo(() -> io.setVoltage(0.0))
                   .withName("Position Mechanism");
    }
    ```

## 7. Simulation and Dual-Vision Strategy
* **All Subsystems MUST be Simulatable:** The AI must always account for WPILib simulation inside `IOSim` implementations (except for Swerve, handled by CTRE).
* **The Dual-Vision Rule:** The robot uses Limelight for real hardware and PhotonVision for simulation.
* **Vision IO Implementations:**
    * `VisionIO.java`: The abstract interface.
    * `VisionIOLimelight.java`: The real implementation used when `Robot.isReal()` is true.
    * `VisionIOPhotonVision.java`: The simulation implementation. Must use `PhotonCamera` and WPILib's `VisionSystemSim`.
* **Instantiation:** In `RobotContainer.java`, the code MUST dynamically inject the correct IO implementation based on `Robot.isReal()`.

---

## 8. Testing & TDD
* Tests must be written in JUnit 5 (`@Test`).
* **No Hardware in Tests:** JUnit tests must NEVER rely on `IOHardware` classes.
* **Mocking:** Tests must define a local `Mock[SubsystemName]IO` class to simulate sensor values.
